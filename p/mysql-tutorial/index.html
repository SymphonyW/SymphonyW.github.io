前言

欢迎来到 MySQL 的世界！无论你是刚接触编程的新手，还是想要补充数据库技能的开发者，这份教程都将为你提供最系统、最详细的学习路径。

MySQL 作为全球最流行的开源关系型数据库，凭借其轻量、高效、跨平台、免费的特性，已经成为 Web 开发、数据分析、后端服务的标配存储方案。从个人博客到大型电商平台，从学生管理系统到金融交易系统，MySQL 都能胜任。

本教程全程基于MySQL 8.0 LTS 稳定版编写，覆盖从环境搭建到生产级运维的全流程，所有代码均可直接复制执行，所有知识点都配有实战案例，确保零基础的你也能一步到位，彻底掌握 MySQL。

第一章：MySQL 基础认知

在动手操作之前，我们先搞懂几个核心概念，帮你建立对 MySQL 的整体认知。

1.1 核心概念

概念 通俗解释
数据库（DB） 存储数据的大仓库，一个 MySQL 服务可以装很多个仓库，比如电商库、学生库
数据表（Table） 仓库里的 Excel 表格，用来分类存储数据，比如用户表、订单表
字段（Column） 表格里的列，比如用户表的id、name、age
记录（Row） 表格里的行，一条完整的数据，比如1,张三,20就是一个用户的记录
SQL 操作数据库的标准语言，就像你和数据库沟通的普通话

1.2 MySQL 整体架构

MySQL 采用经典的客户端 - 服务器架构，整个系统分为三层，理解这个架构能帮你搞懂 SQL 是怎么执行的：

  • 连接层：负责处理客户端的连接请求、权限验证

  • 服务层：核心层，负责 SQL 解析、优化、执行，还有内置函数、权限管理等

  • 存储引擎层：真正负责数据存储和提取的模块，不同引擎有不同的特性

1.3 存储引擎：选对工具事半功倍

MySQL 的存储引擎就像不同的文件管理器，你可以根据业务需求选择：

引擎 特点 适用场景
InnoDB 默认引擎，支持事务、外键、行级锁， crash-safe 99% 的业务场景，生产环境首选
MyISAM 轻量，不支持事务，表级锁，查询快 只读的日志表、静态数据
Memory 数据存在内存里，速度极快，重启就没 临时缓存表、中间计算

💡 提示：不用纠结，直接用默认的 InnoDB 就对了，它能满足你所有的学习和开发需求。

第二章：环境搭建：全平台安装指南

这一章我们来搞定 MySQL 的安装，覆盖 Windows、Linux、Mac 三大平台，每一步都有避坑提示，保证你一次成功。

2.1 Windows 平台安装（图形化界面，新手首选）

Windows 用户推荐用官方的 MSI 安装包，全程图形化操作，不用敲命令。

步骤 1：下载安装包

  1. 打开官网下载页：https://dev.mysql.com/downloads/installer/

  2. 下载MySQL Installer for Windows的完整安装包（约 500M，不要下小的在线包）

步骤 2：安装流程

  1. 双击安装包，选择Developer Default（开发者默认），这个会自动帮你装齐所有需要的组件

  2. 等待依赖安装完成，一路 Next 到配置页面

  3. Type and Networking：保持默认，端口 3306 不要改

  4. 认证方式：保持默认的强密码加密

  5. 设置 root 密码：务必记牢！比如Mysql@2026，要包含大小写、数字、特殊字符

  6. Windows 服务：保持默认，勾选开机自启

  7. 点击 Execute，等待安装完成

步骤 3：配置环境变量（解决 mysql 不是内部命令）

  1. 找到 MySQL 的 bin 目录，默认是C:\Program Files\MySQL\MySQL Server 8.0\bin

  2. 右键此电脑→属性→高级系统设置→环境变量

  3. 找到系统变量里的 Path，双击添加上面的 bin 目录

  4. 重启 CMD，输入mysql -u root -p，输入密码能进入命令行就成功了！

2.4 安装报错急救包

99% 的安装失败都是这些问题，对照着解决：

  1. 1045 错误：密码错误：检查密码大小写，不要复制多余空格

  2. 2003 错误：连不上服务器：MySQL 服务没启动，或者防火墙拦了 3306 端口

  3. 命令找不到：环境变量没配对，重启终端试试

  4. 服务启动失败：之前的 MySQL 没卸载干净，或者路径有中文

建表之前要先搞懂字段的数据类型，这决定了你能存什么数据：

类型 说明 常用场景

DATE 日期 生日、入职日期
DATETIME 日期时间 创建时间、更新时间

IS NULL / IS NOT NULL 空值判断 WHERE phone IS NULL

-- COUNT(*)：统计行数，SUM()：求和，AVG()：平均值，MAX()：最大值，MIN()：最小值

-- 1. 按性别分组，统计每个性别的人数和平均年龄

    COUNT(*) AS 人数, 
    AVG(`age`) AS 平均年龄 
FROM `user`

SELECT `gender`, COUNT(*) AS cnt FROM `user` GROUP BY `gender` HAVING cnt > 1;

第五章：数据完整性与约束：保证数据绝对正确

很多人刚学的时候，会插入乱七八糟的数据，比如手机号重复、年龄负数、订单的用户不存在，约束就是用来解决这个问题的，它是数据库的 ”守门员”，保证你存进去的数据都是合法的。

5.1 约束的核心作用

约束是定义在表上的强制规则，数据库会自动帮你检查，如果你插入 / 更新的数据违反了规则，直接拒绝执行，从根源上避免脏数据。

非空约束是最基础的，它规定这个字段不能为空，你插入数据的时候必须给它赋值。

主键的作用：

  1. 唯一标识一条记录，不会有重复

  2. 其他表的外键要关联它

  3. 默认的主键索引，查询主键的时候速度极快

主键的选择

如果经常纠结主键选什么，这里给你一个最佳实践：

  • 不要用业务字段当主键：比如手机号、身份证号，这些业务字段可能会变

  • 用自增 ID 当主键：INT UNSIGNED NOT NULL AUTO_INCREMENT，简单、高效、不会变

  • UUID 主键：分布式系统用，不过会导致索引碎片，新手不用考虑

第六章：事务管理：保证数据绝对不出错

你有没有遇到过这种情况：转账的时候，A 的钱扣了，B 的钱没加上？这就是没有事务导致的。

事务就是把一组操作打包成一个 ”原子操作”，要么全成功，要么全失败，绝对不会出现只成功一半的情况。

MySQL 中使用 START TRANSACTION 开启事务，COMMIT 提交事务，ROLLBACK 回滚事务。

6.1 事务的 ACID 特性：事务的四大金刚

1. 原子性（Atomicity）：要么全做，要么全不做

原子性是事务最基础的特性，事务里的所有操作，就像一个原子一样，不可分割，不会只做一半。

比如转账的两个操作：A 减 100，B 加 100，这两个必须同时成功，或者同时失败，不能只成功一个。

2. 一致性（Consistency）：数据的状态永远是对的

一致性是说，事务执行前后，数据的完整性约束没有被破坏。

比如转账前，A 有 1000，B 有 1000，总钱数是 2000；转账后，A 有 900，B 有 1100，总钱数还是 2000，这就是一致的。如果 A 减了 100，B 没加，总钱数变成 1900，这就是不一致的，事务会阻止这种情况。

3. 隔离性（Isolation）：多个事务互不干扰

隔离性是说，多个事务同时执行的时候，互相看不到对方的中间状态，不会互相干扰。

比如你和我同时操作同一个账户，我的事务不会影响你的事务，你的也不会影响我的。

4. 持久性（Durability）：提交了就永远不会丢

事务提交之后，数据就永久写到磁盘里了，就算系统断电、崩溃，数据也不会丢。

6.3 并发问题：多个事务同时操作会出什么问题？

如果只有一个事务，那没问题，但是实际生产中，几百个事务同时执行，就会出现并发问题，主要有三个：

1. 脏读：读到了别人未提交的数据

脏读就是，一个事务读到了另一个事务还没提交的数据，然后那个事务回滚了，你读到的就是脏数据。

举个例子：

  • 事务 1：A 给 B 转 100，A 减了 100，还没提交

  • 事务 2：查 A 的余额，读到了 900

  • 事务 1：出错了，回滚，A 的余额变回 1000

  • 结果：事务 2 读到的 900 就是脏数据，根本不存在

2. 不可重复读：同一个事务里，两次读同一个数据，结果不一样

不可重复读就是，在同一个事务里，你两次读同一个数据，结果不一样，因为中间被别的事务改了。

举个例子：

  • 事务 1：查 A 的余额，读到 1000

  • 事务 2：改 A 的余额，改成 900，提交了

  • 事务 1：再查 A 的余额，读到 900

  • 结果：同一个事务里，两次读同一个数据，结果不一样，这就是不可重复读

3. 幻读：同一个事务里，两次查询，行数不一样

幻读就是，在同一个事务里，你两次查同一个范围的数据，第二次多了一行，就像幻觉一样。

举个例子：

  • 事务 1：查年龄 > 20 的用户，查到 2 行

  • 事务 2：插入了一个年龄 25 的用户，提交了

  • 事务 1：再查年龄 > 20 的用户，查到 3 行

  • 结果：同一个事务里，两次查一样的条件，结果行数不一样，这就是幻读

6.4 事务隔离级别：解决并发问题

为了解决这些并发问题，SQL 定义了四个隔离级别，级别越高，并发问题越少，但是性能越差。

隔离级别 脏读 不可重复读 幻读 说明
读未提交（Read Uncommitted） ✅ ✅ ✅ 最低，基本不用，性能最好
读已提交（Read Committed） ❌ ✅ ✅ Oracle 默认，解决了脏读
可重复读（Repeatable Read） ❌ ❌ ✅ MySQL 默认，解决了脏读、不可重复读
串行化（Serializable） ❌ ❌ ❌ 最高，所有问题都解决，但是性能最差，完全串行

💡 提示：MySQL 默认的可重复读，通过 MVCC（多版本并发控制），其实也解决了幻读，所以你不用改隔离级别，默认的就够用了。

6.5 事务避坑

  1. 不要开长事务：事务不要开太久，不然会锁表，影响其他业务，比如你开了事务，然后去调用接口，半天不提交，这就会出问题

  2. 异常一定要回滚：代码里如果出错了，一定要调用 ROLLBACK，不然事务会一直挂着

  3. 不要在事务里做远程调用：远程调用很慢，会把事务拖长

  4. 不要用事务做批量导入：大的批量操作，拆成小事务，不然锁太久了

第七章：索引：让查询快 1000 倍的秘密

很多人刚学的时候，数据少，查询很快，等数据到了 100 万，查询就慢的要死，这就是因为没加索引！

索引就是把数据按照某个字段进行排序，建立一个快速查找的目录，就像书的目录一样，有了索引，你就不用一页一页翻书了，直接就能找到你要的内容，查询速度能提升几百上千倍！

7.1 索引的原理：B + 树到底是什么？

MySQL 默认的索引是B + 树，这是一种有序的平衡树，所有的数据都存在叶子节点，而且叶子节点是有序的，用链表连起来。

B + 树的特点：

  1. 有序：所有数据都是排好序的，所以范围查询、排序都特别快

  2. 查询快：查询的时间复杂度是 O (log n)，100 万的数据，只要查 20 次就找到了

  3. 叶子节点有序：叶子节点是用链表连起来的，所以遍历全表也很快

7.2 索引的类型：不同的索引用在不同的地方

类型 作用 适用场景
主键索引 主键自带的，聚簇索引，每个表必须有 主键 ID，查询 ID 的时候
二级索引 普通的索引，叶子节点存主键 普通的查询字段，比如 name
唯一索引 唯一字段的索引，值不能重复 手机号、用户名、订单号
复合索引 多个字段的索引，比如 (name, age) 经常一起查询的多个字段
覆盖索引 查询的字段都在索引里，不用回表 只查几个字段的查询
全文索引 文本的模糊搜索，比如文章内容 全文检索，比如搜索文章

7.3 回表：什么是回表，怎么避免？

二级索引的叶子节点存的是数据的主键，所以你用二级索引查到主键之后，还要用主键去主键索引里查完整的数据，这就叫回表，回表会多一次 IO，所以会慢一点。

如果你的查询的字段，都在索引里，那就不用回表了，这就是覆盖索引，速度会快很多。

比如：

1
2
3
4
5
6

-- 我们有复合索引(name, age)
-- 这个查询，只查name和age，都在索引里，不用回表，覆盖索引
SELECT `name`, `age` FROM `user` WHERE `name` = '张三';

-- 这个查询，要查phone，phone不在索引里，所以要回表
SELECT `name`, `age`, `phone` FROM `user` WHERE `name` = '张三';

7.4 最左前缀原则：复合索引的核心规则

复合索引是多个字段的索引，比如(a,b,c)，它的顺序很重要，查询的时候，必须从最左边的字段开始匹配，这就是最左前缀原则。

举个例子，我们有复合索引(name, age, gender)：

查询条件 能不能用到索引？
WHERE name = ‘张三’ ✅ 能，匹配最左的 name
WHERE name = ‘张三’ AND age = 20 ✅ 能，匹配 name’+age
WHERE name = ‘张三’ AND age = 20 AND gender = ‘男’ ✅ 能，匹配全部
WHERE age = 20 AND gender = ‘男’ ❌ 不能，跳过了最左的 name
WHERE name = ‘张三’ AND gender = ‘男’ ✅ 能，name 能用到，gender 用不到

💡 所以建复合索引的时候，要把查询频繁的、区分度高的字段放最左边，把范围查询的字段放右边。

7.6 索引失效：最容易踩的坑，加了索引也没用

很多人加了索引，但是查询还是很慢，因为索引失效了，也就是你写的 SQL，导致索引用不了，变成了全表扫描。

我们一个个来看，这些都是最容易踩的坑：

1. 对索引字段做函数操作

如果你对索引字段做函数操作，MySQL 就没法用索引了，因为索引存的是原始值，不是函数处理后的值。

反例（索引失效）：

1
2

-- 对create_time做了YEAR函数，索引失效
SELECT * FROM user WHERE YEAR(create_time) = 2026;

正例（索引生效）：

1
2

-- 用范围查询，不用函数，索引生效
SELECT * FROM user WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01';

2. 模糊查询，开头用通配符

LIKE '%xxx'，开头的通配符，会导致索引失效，因为索引是有序的，开头不确定的话，没法匹配。

反例（索引失效）：

1
2

-- 开头的%，索引失效
SELECT * FROM user WHERE name LIKE '%张三';

正例（索引生效）：

1
2

-- 结尾的%，索引生效
SELECT * FROM user WHERE name LIKE '张三%';

3. 用 OR 连接没有索引的字段

如果 OR 的两个字段，一个有索引，一个没有，那整个查询都不会用索引。

反例（索引失效）：

1
2

-- name有索引，age没有，OR导致整个索引失效
SELECT * FROM user WHERE name = '张三' OR age = 20;

4. 隐式类型转换

如果字段是字符串，你传了数字，MySQL 会自动做类型转换，导致索引失效。

反例（索引失效）：

1
2

-- phone是字符串，你传了数字13800138001，隐式转换，索引失效
SELECT * FROM user WHERE phone = 13800138001;

正例（索引生效）：

1
2

-- 传字符串，索引生效
SELECT * FROM user WHERE phone = '13800138001';

5. 用 NOT、’!=、<>、IS NOT NULL

这些操作，会导致索引失效，因为 MySQL 觉得全表扫描更快。

7.7 怎么看索引有没有生效：Explain 执行计划

很多人加了索引，不知道有没有生效，这时候就可以用EXPLAIN，它能告诉你 SQL 的执行计划，索引有没有用到。

用法很简单，在 SQL 前面加 EXPLAIN：

1

EXPLAIN SELECT * FROM user WHERE name = '张三';

然后看结果里的type字段，这个是查询的类型，越好的类型，速度越快：

  • system > const > eq_ref > ref > range > index > ALL

如果type是ALL，那就是全表扫描，索引没生效！ 如果是ref、range这些，就是索引生效了。

还有key字段，显示的是用到的索引，如果这个是空的，就是没用到索引。

7.8 索引使用的最佳实践

  1. 不要给小表加索引：数据就几百条，全表扫描比索引还快

  2. 不要给区分度低的字段加索引：比如性别，只有男 / 女，加了索引也没用

  3. 不要给频繁修改的字段加索引：修改数据的时候，要同步修改索引，会变慢

  4. 索引不是越多越好：每个索引都会占用空间，而且会拖慢增删改的速度，一个表的索引不要超过 5 个

  5. 慢查询优化：开启慢查询日志，找到慢 SQL，然后给它们加索引

第十一章：必避的 10 个坑

最后，总结一下最容易踩的坑，帮你少走弯路：

  1. UPDATE/DELETE 忘记加 WHERE：这是最常见的，一不小心全表数据没了，执行前先 SELECT 看看对不对

  2. 字符集乱码：建库建表一定要用 utf8mb4，不要用 latin1，备份恢复也要指定字符集

  3. 密码忘记：root 密码一定要记牢，重置密码很麻烦

  4. 外键导致删不了表：有外键关联的话，要先删从表，或者关掉外键校验

  5. 大表 SELECT：大表查所有字段，会把数据库卡死，要啥查啥，加 LIMIT

  6. 长事务：事务不要开太久，会锁表，影响业务

  7. 索引建太多：索引不是越多越好，会拖慢增删改的速度

  8. 备份不验证：备份完了一定要看看备份文件有没有内容，别备份了个空的

  9. root 用户到处用：生产环境不要用 root，要创建专用用户，最小权限原则

  10. 路径有中文：MySQL 对中文路径兼容性差，安装、备份的路径都用英文
